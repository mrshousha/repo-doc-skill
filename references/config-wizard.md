# Config Wizard (`/repo-doc config` command)

This is the first command any engineer runs on a new repo. It produces or updates
`repodoc.yaml`. It must be safely re-runnable at any time.

---

## 1.1 — Detect Re-run

Check whether `repodoc.yaml` already exists.

**First run:** proceed through the full wizard (sections 1.2–1.8 below).

**Re-run:** load the current config, display it, and present a numbered change menu:

```
Current configuration for <service.name>:

  1.  Service name:        payment-service
  2.  Description:         Handles payment processing and settlement
  3.  Owner:               #team-payments
  4.  Jira enabled:        true
  5.  Jira ticket pattern: [A-Z]+-[0-9]+
  6.  Multi-repo enabled:  true
  7.  GitHub org:           deliveryhero
  8.  Docs output dir:     docs/
  9.  Audit on commit:     false
  10. Audit on PR:         false
  11. PR template:         enabled
  12. Exclude dirs:        node_modules, vendor, .git, dist, build, __pycache__, .gradle, target
  13. Max questions/run:   20

Which numbers would you like to change? (e.g. "3, 5" or "none")
```

For each selected number:
- Show the current value
- Explain consequences of changing if relevant
- Ask for the new value

After all changes are collected, show a diff summary and confirm:

```
Changes to be written:
  owner: "#team-payments" → "#team-payments-eu"

Write these changes? (y/n)
```

Display the MCP connection status from the pre-flight probe.

If connected:
```
MCP Connection Status:
  ✅ Atlassian (Jira) — connected

Jira enrichment will be active. This significantly improves the quality
of the "why" sections in your architecture and feature documentation.
```

If not connected:
```
MCP Connection Status:
  ❌ Atlassian (Jira) — not connected

  Jira enrichment will be skipped. You can connect it later via your
  AI agent's settings → Integrations and re-run /repo-doc config to enable it.
```

Do **not** block the wizard if Jira is unavailable — the engineer can enable it later.

## 1.3 — Topology Detection

Run a skeleton scan (directory tree only, no file content reads). Detect:
- **Single service**, **monorepo** (multiple package manifests / Dockerfiles / main
  entrypoints), or **library**
- **Primary language(s)** by file extension distribution

Report findings and ask for confirmation:

```
Detected: monorepo with 3 apparent services

  1. payment-service    (src/payment/)
  2. notification-service (src/notifications/)
  3. shared-lib         (src/shared/)

Primary language: Java (87%), YAML (8%), SQL (5%)

Document all services, or select specific ones? (all / 1,2 / etc.)
```

For selected services: docs under `docs/<service-name>/`.
For a single service: docs under `docs/` directly.

### Docs Output Directory Conflict Detection

After topology detection, check if a `docs/` directory already exists **with
non-repodoc content** (i.e. files not generated by repo-doc, no sentinel comments).

If it does, prompt **two questions**:

**Question 1 — Use existing content as reference:**

```
A docs/ directory already exists with existing content.

Would you like repo-doc to use the existing docs as reference material
when generating documentation? This can improve accuracy by incorporating
any manually written context, architecture notes, or onboarding guides
that already exist.

  1. Yes — read existing docs and use them as context (Recommended)
  2. No — generate fresh documentation from code only

Select: _
```

If the user selects **Yes**, store `docs_site.seed_from_existing: true` in
`repodoc.yaml`. During the scanning phase (`scanner.md`), the existing docs
directory should be read as **additional context** (not as source code) — treat
its content as human-written notes that can inform the generated documentation.
Do not copy content verbatim; use it to fill gaps, validate assumptions, and
improve the "why" sections.

If the user selects **No**, set `docs_site.seed_from_existing: false` and skip
reading the existing docs directory entirely.

**Question 2 — Output directory:**

```
Where should repo-doc write the generated documentation?

  1. docs-gen/ (Recommended — keeps existing docs/ untouched)
  2. docs/ (overwrite existing directory)
  3. Custom path

Select: _
```

Store the choice in `docs_site.output_dir`. If the user picks a custom path,
validate it's a relative path and doesn't conflict with source directories.

If no `docs/` directory exists, default to `docs/` silently without prompting
either question.

## 1.4 — Service Identity

Ask:
- **Service name** — auto-suggest from repo name, directory name, or `name` field in
  package manifest (package.json, pom.xml, build.gradle, go.mod, Cargo.toml, etc.)
- **Description** — one sentence describing what it does (used in `docs/index.md` and
  `docs/onboarding.md`)
- **Owner** — team name or Slack channel (used in onboarding docs)

## 1.5 — Jira Configuration

**Always ask for the ticket pattern**, regardless of MCP connection status. The
pattern is used for git commit/PR scanning even when Jira MCP is not connected.

```
What ticket ID format does your team use?
Examples: IE-1234, CORE-567, PAY-89
Default regex [A-Z]+-[0-9]+ covers most formats. Press Enter to accept.
```

If the user provides a specific pattern (e.g. "we use IE-XXXX"), derive the regex
from their example and store it in `jira.ticket_pattern`.

If Atlassian MCP is connected: set `jira.enabled: true`.
If not connected: set `jira.enabled: false` (ticket pattern is still stored for
git history scanning — see `enrichment.md` §5.1).

## 1.9 — Write `repodoc.yaml`

Show a preview of the full config and confirm before writing.

### Full Schema

```yaml
# repodoc.yaml — generated by /repo-doc config
# Re-run /repo-doc config to change any of these settings.
version: 1

service:
  name: ""                          # Required
  description: ""
  owner: ""

jira:
  enabled: true                     # false if Atlassian MCP not connected
  ticket_pattern: "[A-Z]+-[0-9]+"

multi_repo:
  enabled: false
  auto_discover: true
  github_org: ""
  known_services: []

docs_site:
  enabled: true
  framework: "mkdocs-material"
  output_dir: "docs/"
  site_name: ""                     # Auto-derived from service.name if empty
  seed_from_existing: false         # true if user opted to use existing docs/ as reference

audit:
  on_commit: false
  manual: true

diagrams:
  data_flow: true
  sequence: true
  architecture: true
  entity_model: true
  format: "mermaid"

scan:
  exclude_dirs:
    - node_modules
    - vendor
    - .git
    - dist
    - build
    - __pycache__
    - .gradle
    - target
    - docs                          # Must match docs_site.output_dir — prevents circular scanning
  exclude_patterns:
    - "*.lock"
    - "*.pb.go"
    - "*_generated.go"
    - "*.min.js"
  skip_binary_files: true

questions:
  max_per_run: 20

pr_template:
  enabled: true
```

### Dynamic Scan Exclusion

The `scan.exclude_dirs` list must always include the docs output directory. When
writing `repodoc.yaml`:
- If `docs_site.output_dir` is `docs/`, the default exclusion (`docs`) is correct.
- If `docs_site.output_dir` is something else (e.g. `docs-gen/`), replace the `docs`
  entry with the actual directory name (e.g. `docs-gen`). Both must stay excluded
  to prevent circular scanning.

### .gitignore Updates

After writing `repodoc.yaml`, update `.gitignore`:
- **Add** `repodoc-audit-report.md` (generated on each audit, not committed)
- **Add** `.repodoc-backup/` (auto-backups should not be committed)
- **Ensure NOT listed:** `.repodoc-state.json` (must be committed — shared team state)

If `.gitignore` doesn't exist, create it with these entries.

### Completion Message

After writing both files, print:

```
Configuration saved to repodoc.yaml

Next step: run /repo-doc init to scan the codebase and generate documentation.

This will:
  • Scan your code and build architecture diagrams
  • Generate a full docs/ site with MkDocs Material
  • Create an onboarding guide, API reference, security docs, and more
  • Set up a local docs server you can browse at http://localhost:8000
```
